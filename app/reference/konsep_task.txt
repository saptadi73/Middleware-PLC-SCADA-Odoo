Konsep task middleware (VERSI AKTUAL)

Tanggal update: 2026-03-06

============================================================
RINGKASAN ARSITEKTUR
============================================================
- READ area PLC (feedback proses): D6000-D6977
  - BATCH_READ_01..10
  - status_read_data per-batch: D6076, D6176, ..., D6976
- WRITE area PLC (command dari middleware): D7000-D7976
  - WRITE_BATCH01..10
  - handshake write area utama: D7076
- Equipment failure: D8000-D8022
- Manual weighing: D9000-D9013

Equipment target produksi:
- 13 silo: ID 101..113
- 2 liquid: LQ114 (tetes), LQ115 (fml)

============================================================
ATURAN STATUS (PENTING)
============================================================
1) Di WRITE area, middleware SELALU menulis default:
   - status_manufacturing = 0
   - status_operation = 0
   - status_read_data = 0

2) Middleware TIDAK mengubah status proses di WRITE menjadi 1.
   PLC yang berwenang mengubah status proses selama eksekusi batch.

3) Middleware hanya menandai status_read_data=1 di READ area
   untuk batch yang SUDAH selesai (status_manufacturing=1) saat dibaca.

============================================================
TASK 1 - AUTO SYNC ODOO -> DB -> PLC WRITE
============================================================
Tujuan:
- Ambil daftar MO dari Odoo
- Simpan ke mo_batch
- Tulis ke PLC WRITE area

Alur:
1. Cek mo_batch kosong/tidak.
   - Jika tidak kosong: skip (tunggu batch aktif selesai).
2. Fetch MO list dari Odoo.
3. Stage ke mo_batch (deferred commit).
4. Write queue ke PLC WRITE area (slot batch 1..N).
5. Commit DB hanya jika write sukses penuh.
6. Handshake D7076 dipakai untuk mencegah overwrite data belum dibaca PLC.

Catatan:
- Field mapping WRITE mengikuti MASTER_BATCH_REFERENCE.json.
- Status default di WRITE dipaksa 0 oleh middleware.

============================================================
TASK 2 - PLC READ -> UPDATE mo_batch
============================================================
Tujuan:
- Baca feedback aktual dari READ area PLC
- Update actual consumption / actual weight / status ke mo_batch

Alur:
1. Baca semua batch READ 01..10.
2. Match ke mo_batch berdasarkan (batch_no, mo_id), lalu fallback aman.
3. Update field actual jika ada perubahan dan batch DB belum completed.
4. Untuk batch READ dengan status_manufacturing=1:
   - mark status_read_data READ area batch tersebut = 1.
5. Untuk batch belum selesai (status_manufacturing!=1):
   - jangan set status_read_data READ area.

Catatan:
- Task 2 tidak melakukan archive.
- Guard numerik dipakai untuk skip nilai PLC yang implausible.

============================================================
TASK 3 - PROCESS COMPLETED (ODOO SYNC + ARCHIVE)
============================================================
Filter:
- status_manufacturing=1
- update_odoo=false

Alur:
1. Kirim konsumsi aktual ke Odoo.
2. Jika sukses penuh:
   - archive ke mo_histories
   - hapus dari mo_batch (atomic flow)
3. Jika gagal/partial:
   - batch tetap di mo_batch
   - akan retry di cycle Task 3 berikutnya

Prinsip safety:
- Tidak menghapus batch dari mo_batch sebelum Odoo sinkron sukses.

============================================================
TASK 4/5/6 (SINGKAT)
============================================================
Task 4:
- Monitoring kesehatan batch aktif (stuck/warning).

Task 5:
- Equipment failure monitoring (PLC -> DB -> Odoo).

Task 6:
- Cleanup system log periodik.
